That collapsed almost all the unattended-execution complexity — since the continuation always resumes a session that's already interactively open, the existing approval widget and tool pipeline apply unchanged.
  - **`scheduleWakeup`** (`packages/shared/src/schemas.ts`, BUILD mode only): `{ delaySeconds (10–1800), reason, prompt }`. `reason` is the short label shown in the transcript; `prompt` is the exact continuation instruction the model writes for its future self, replayed verbatim on fire (not a generic placeholder). Dispatched specially in `use-chat.ts`'s `onToolCall` (like `spawnAgent`/`askUser`/`switchMode`, bypassing `executeLocalTool`): a module-level `_pendingWakeups` map (sessionId → timeout handle, alongside the existing `_activeModes` map) registers a `setTimeout`; firing always pushes onto the existing message queue (`setMessageQueue`) rather than calling `chat.sendMessage`/reading `chat.status` directly — that closure is captured at schedule time and can be many renders stale by fire time (up to 30 min later), whereas `setMessageQueue` is a plain state setter safe to call regardless of closure age, and the pre-existing auto-drain effect already sends queued messages once `chat.status` is actually `"ready"`. Cancelled (cleared) on session unmount and at the top of `submit()` — a real user message always supersedes a pending wakeup. Inline transcript rendering added to `bot-message.tsx`.

Caught by the user asking "doesn't the volatile spoil the whole thing?" before this shipped anywhere live.

Skipped when the last message is a tool-result continuation (`role: "tool"`, no text-part slot in `ToolContent`) — accepted trade-off, not a bug.

**Bug found via that manual testing, now fixed:** the `TextStep` submit wiring initially mirrored `EditKeyView`'s imperative `input.onSubmit = fn` pattern, but that only works for `<textarea>` — pressing Enter on the name field did nothing. Root cause: `InputRenderable.submit()` *overrides* `TextareaRenderable.submit()` to `emit("enter", value)` instead of invoking `._submitListener` (the mechanism `.onSubmit = fn` relies on), so the assignment was silently inert for `<input>`. The declarative `onSubmit` JSX prop looked like the fix, but its type is an unusable intersection of two incompatible library-internal signatures (`(event: SubmitEvent) => void` from the inherited Textarea-core option, `(value: string) => void` from the React-only Input variant) — confirmed by reading `@opentui/react`'s reconciler source (`chunk-dz5xbr9p.js`), which does correctly map that prop to `InputRenderableEvents.ENTER` at runtime, but `tsc` rejects passing a plain string-typed handler to it. Root cause: `ProviderFieldsWizard`/`ModelFieldsWizard` render one `<TextStep>` per `if (step === ...)` branch, all at the same JSX tree position with the same component type — React doesn't remount between steps, it just re-renders the same `TextStep` instance (and its underlying `<input>` renderable) with new props. ## Architecture Decisions

- **Always slice before sending to the model** — Any server route that sends message history to an LLM must first call `getLastBoundaryIndex(messageRecords)` and slice from that index + 1. This ensures the model only sees the current context window (post-clear or post-compact). Applied consistently in `chat.ts` (live stream), `sessions.ts` compact, and `sessions.ts` handoff. `getLastBoundaryIndex` lives in `lib/helpers.ts`.

- Bun workspaces monorepo — four packages: cli, server, database, shared.
- OpenTUI + React 19 for terminal rendering (not Ink).
- Tool calls executed client-side in the CLI; server only orchestrates the AI turn.
- Provider resolution is model-driven: the model the user selects determines which key is used. Direct provider keys (Anthropic, OpenAI, Gemini) take priority when the model maps to that provider; OpenRouter is the fallback for any model without a matching direct key.
- Config stored globally at `~/.koincode/config.json` — holds provider keys and user preferences, shared across all projects.
- Project-specific config stored at `.koincode/config.json` — holds project-level hooks and permissions.
- Hooks support both global and project scopes: global hooks apply to all projects, project hooks override global hooks for the current project.
- Message history stored as a JSON blob on the Session row rather than normalized message rows.
- PLAN mode and BUILD mode share the same session; mode is selected at session creation and controls which tools are exposed to the model.
