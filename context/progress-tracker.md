**Bug found via that manual testing, now fixed:** the `TextStep` submit wiring initially mirrored `EditKeyView`'s imperative `input.onSubmit = fn` pattern, but that only works for `<textarea>` ‚Äî pressing Enter on the name field did nothing. Root cause: `InputRenderable.submit()` *overrides* `TextareaRenderable.submit()` to `emit("enter", value)` instead of invoking `._submitListener` (the mechanism `.onSubmit = fn` relies on), so the assignment was silently inert for `<input>`. The declarative `onSubmit` JSX prop looked like the fix, but its type is an unusable intersection of two incompatible library-internal signatures (`(event: SubmitEvent) => void` from the inherited Textarea-core option, `(value: string) => void` from the React-only Input variant) ‚Äî confirmed by reading `@opentui/react`'s reconciler source (`chunk-dz5xbr9p.js`), which does correctly map that prop to `InputRenderableEvents.ENTER` at runtime, but `tsc` rejects passing a plain string-typed handler to it. Fixed by subscribing directly via `inputRef.current.on(InputRenderableEvents.ENTER, handler)` / `.off(...)` in a `useEffect`, bypassing both the broken imperative property and the broken prop type.

**Second bug found via manual testing, now fixed:** after the Enter-to-advance fix, text typed into one step (e.g. provider name) was carrying over into the next step's input (base URL) ‚Äî had to be manually deleted before typing the new value. Root cause: `ProviderFieldsWizard`/`ModelFieldsWizard` render one `<TextStep>` per `if (step === ...)` branch, all at the same JSX tree position with the same component type ‚Äî React doesn't remount between steps, it just re-renders the same `TextStep` instance (and its underlying `<input>` renderable) with new props. The `setText(initialValue)` effect meant to seed each field only re-runs when `initialValue` actually changes by value, and since most fields start `""`, going from one empty field to the next empty field never re-triggered it ‚Äî so the previous field's typed text stayed in the live buffer. Fixed by giving every `<TextStep>`/`<BooleanStep>` return a distinct `key` (`"name"`, `"baseURL"`, `"apiKey"`, `"modelId"`, `"contextWindow"`, `"vision"`), forcing React to unmount/remount a fresh instance on every step transition.

**Third bug found via manual testing, now fixed:** selecting a custom model showed the opaque internal id (e.g. `custom/1a2b3c`) in the status bar instead of the real model name ‚Äî correct for routing (`resolveChatModel` needs the opaque id to look up the right provider) but never meant to be user-visible. ## Architecture Decisions

- **Always slice before sending to the model** ‚Äî Any server route that sends message history to an LLM must first call `getLastBoundaryIndex(messageRecords)` and slice from that index + 1. This ensures the model only sees the current context window (post-clear or post-compact). Applied consistently in `chat.ts` (live stream), `sessions.ts` compact, and `sessions.ts` handoff. `getLastBoundaryIndex` lives in `lib/helpers.ts`.

- Bun workspaces monorepo ‚Äî four packages: cli, server, database, shared.
- OpenTUI + React 19 for terminal rendering (not Ink).
- Tool calls executed client-side in the CLI; server only orchestrates the AI turn.
- Provider resolution is model-driven: the model the user selects determines which key is used. Direct provider keys (Anthropic, OpenAI, Gemini) take priority when the model maps to that provider; OpenRouter is the fallback for any model without a matching direct key.
- Config stored globally at `~/.koincode/config.json` ‚Äî holds provider keys and user preferences, shared across all projects.
- Project-specific config stored at `.koincode/config.json` ‚Äî holds project-level hooks and permissions.
- Hooks support both global and project scopes: global hooks apply to all projects, project hooks override global hooks for the current project.
- Message history stored as a JSON blob on the Session row rather than normalized message rows.
- PLAN mode and BUILD mode share the same session; mode is selected at session creation and controls which tools are exposed to the model.
