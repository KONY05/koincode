## Recently Completed (Phase 2 — continued)

- **Prompt caching (Anthropic direct + via OpenRouter)** — Cost-optimization feature: since chat APIs are stateless, every turn re-sends the full system prompt, tools, and history, so cost compounds as a session grows. Implemented explicit `cache_control` breakpoints for the direct-Anthropic-key path, plus a separate, simpler mechanism for Anthropic models routed through OpenRouter (Google's context-caching lifecycle and custom/Ollama providers remain out of scope — see spec for why). Went through two design corrections on the direct-Anthropic path before landing on the right shape — worth reading in full in the spec's Status section, since the wrong version would have looked like it worked (typechecks fine) while silently never hitting cache on the one breakpoint that mattered:
  1. First tried reordering the system prompt's volatile `ideActiveFile` section (changes turn-to-turn as the user switches editor tabs) to the end of the joined string — insufficient, since Anthropic's `cache_control` invalidates a whole content block on any change regardless of position within it.
  2. Then split system into two separate `SystemModelMessage`s (`stable` cached, `volatile` uncached) — still wrong, because Anthropic's caching is a *cumulative prefix* across the fixed canonical order `tools → system → messages` (confirmed against Anthropic's own docs, not assumed): a breakpoint's hit depends on everything before it in that order, so `ideActiveFile` sitting anywhere in `system` — split out or not — still sat upstream of the `messages`/history breakpoint and silently spoiled it (the `tools` breakpoint, canonically first, was never actually at risk). Caught by the user asking "doesn't the volatile spoil the whole thing?" before this shipped anywhere live.
  3. **Final fix:** `ideActiveFile` removed from the system prompt entirely — `buildSystemPrompt` (`packages/server/src/prompts/system-prompt.ts`) is back to one fully-stable string, no split. New `appendIdeContext` (`packages/server/src/lib/prompt-caching.ts`) appends the IDE-context text directly onto the *newest message's* own content instead — that message is already a fresh, uncached write every turn, so attaching volatile text there is free, and once finalized it never threatens future turns' cache hits. Skipped when the last message is a tool-result continuation (`role: "tool"`, no text-part slot in `ToolContent`) — accepted trade-off, not a bug.

  Remaining pieces: `packages/server/src/lib/prompt-caching.ts` also has `buildCachedSystemMessage` (single cached system message now, no volatile counterpart), `withToolsCacheControl` (marks the last tool definition, closing `tools` as one cached prefix), and `withHistoryCacheControl` (marks the newest message every turn — now actually reliable, since nothing volatile sits upstream of it anymore; `appendIdeContext` runs first so the breakpoint's hash covers the final content). `ResolvedModel` (`packages/server/src/lib/models.ts`) gained `promptCaching?: boolean`, set `true` only in `resolveAnthropicModel`'s direct-key branch — deliberately not inferred from `provider === "anthropic"` alone, since that label is also returned on the OpenRouter-fallback path, which uses an entirely separate mechanism (below), not `chat.ts`'s manual breakpoint helpers. `packages/cli/src/lib/cost.ts`'s `estimateSessionCost` updated to price `usage.inputTokenDetails.{cacheReadTokens,cacheWriteTokens,noCacheTokens}` at their real rates (read ≈0.1× input rate, write ≈1.25×) instead of billing all input tokens at the flat rate — falls back to the exact prior flat-rate formula when a provider doesn't report the breakdown. `/agent-step` (single-step, ephemeral, no accumulating history) intentionally left untouched — out of scope for this pass.

  **Correction, same session:** OpenRouter pass-through was initially written off in the spec as "unresolved, don't commit to it" — wrong, the user pushed back ("openrouter implements prompt caching for providers that support it") and it checked out against both OpenRouter's own docs and the installed `@openrouter/ai-sdk-provider@2.9.0` type declarations (`OpenRouterChatSettings.cache_control: { type: "ephemeral", ttl?: "5m"|"1h" }`, confirmed in `node_modules`). OpenAI-via-OpenRouter caches automatically with no config, same as calling OpenAI directly. Anthropic-via-OpenRouter gets a top-level `cache_control` setting passed once at model-construction time — OpenRouter auto-advances the breakpoint itself as the conversation grows, so none of `chat.ts`'s manual per-turn breakpoint bookkeeping is needed for this path. Implemented entirely inside `resolveViaOpenRouter` (`packages/server/src/lib/models.ts`): `openrouter.chat(routerModelId, settings)` now passes `{ cache_control: { type: "ephemeral" } }` when `provider === "anthropic"` — exact gate, not a heuristic, since the app's `"openrouter"`-native model list has no `anthropic/*` entries today. Zero changes to `chat.ts` needed for this path.

  Spec: `context/feature-specs/36-prompt-caching-support.md`. **Not verified in-agent:** OpenAI's automatic caching actually landing, and Anthropic's cache hit rate in a live session (direct or via OpenRouter) — all need a real provider account to confirm; `bun run typecheck` is clean but no live request was made.

**Bug found via that manual testing, now fixed:** the `TextStep` submit wiring initially mirrored `EditKeyView`'s imperative `input.onSubmit = fn` pattern, but that only works for `<textarea>` — pressing Enter on the name field did nothing. Root cause: `InputRenderable.submit()` *overrides* `TextareaRenderable.submit()` to `emit("enter", value)` instead of invoking `._submitListener` (the mechanism `.onSubmit = fn` relies on), so the assignment was silently inert for `<input>`. The declarative `onSubmit` JSX prop looked like the fix, but its type is an unusable intersection of two incompatible library-internal signatures (`(event: SubmitEvent) => void` from the inherited Textarea-core option, `(value: string) => void` from the React-only Input variant) — confirmed by reading `@opentui/react`'s reconciler source (`chunk-dz5xbr9p.js`), which does correctly map that prop to `InputRenderableEvents.ENTER` at runtime, but `tsc` rejects passing a plain string-typed handler to it. Fixed by subscribing directly via `inputRef.current.on(InputRenderableEvents.ENTER, handler)` / `.off(...)` in a `useEffect`, bypassing both the broken imperative property and the broken prop type.

**Second bug found via manual testing, now fixed:** after the Enter-to-advance fix, text typed into one step (e.g. provider name) was carrying over into the next step's input (base URL) — had to be manually deleted before typing the new value. Root cause: `ProviderFieldsWizard`/`ModelFieldsWizard` render one `<TextStep>` per `if (step === ...)` branch, all at the same JSX tree position with the same component type — React doesn't remount between steps, it just re-renders the same `TextStep` instance (and its underlying `<input>` renderable) with new props. The `setText(initialValue)` effect meant to seed each field only re-runs when `initialValue` actually changes by value, and since most fields start `""`, going from one empty field to the next empty field never re-triggered it — so the previous field's typed text stayed in the live buffer. Fixed by giving every `<TextStep>`/`<BooleanStep>` return a distinct `key` (`"name"`, `"baseURL"`, `"apiKey"`, `"modelId"`, `"contextWindow"`, `"vision"`), forcing React to unmount/remount a fresh instance on every step transition.

Applied at all 4 render sites that print a bare model-id string to the user: `status-bar.tsx` (current session model), `messages/bot-message.tsx` (per-message historical model, can't reuse the live "current model" value since old messages may reference a model since changed or deleted), and `dialogs/context-dialog.tsx` (×3 render branches). `usage.ts`/`analytics.ts` were checked at the time and didn't need it, since they only used the model id as a lookup/tracking key — see below, this changed shortly after.

## Architecture Decisions

- **Always slice before sending to the model** — Any server route that sends message history to an LLM must first call `getLastBoundaryIndex(messageRecords)` and slice from that index + 1. This ensures the model only sees the current context window (post-clear or post-compact). Applied consistently in `chat.ts` (live stream), `sessions.ts` compact, and `sessions.ts` handoff. `getLastBoundaryIndex` lives in `lib/helpers.ts`.

- Bun workspaces monorepo — four packages: cli, server, database, shared.
- OpenTUI + React 19 for terminal rendering (not Ink).
- Tool calls executed client-side in the CLI; server only orchestrates the AI turn.
- Provider resolution is model-driven: the model the user selects determines which key is used. Direct provider keys (Anthropic, OpenAI, Gemini) take priority when the model maps to that provider; OpenRouter is the fallback for any model without a matching direct key.
- Config stored globally at `~/.koincode/config.json` — holds provider keys and user preferences, shared across all projects.
- Project-specific config stored at `.koincode/config.json` — holds project-level hooks and permissions.
- Hooks support both global and project scopes: global hooks apply to all projects, project hooks override global hooks for the current project.
- Message history stored as a JSON blob on the Session row rather than normalized message rows.
- PLAN mode and BUILD mode share the same session; mode is selected at session creation and controls which tools are exposed to the model.
