# Feature 47: AGENTS.md Content Injection

## Origin

While testing how influential AGENTS.md actually is, the user found it isn't followed reliably — the model only picks up its instructions when it happens to decide to read the file via a tool call. A conversational message ("hey, how are you?") with a project's `AGENTS.md` open didn't trigger any read, and the file's instructions had zero effect on the response. By contrast, an opencode demo video showed the model obeying its `AGENTS.md` without any visible read step, implying opencode loads the file's actual content into context rather than just telling the model such a file might exist.

## Current state

`getAgentsMdSection()` in `packages/server/src/prompts/system-prompt.ts:124-134` is the entire mechanism today. It does not read or transmit any file content — it's a fixed block of prose in the system prompt telling the model "repos often contain AGENTS.md files... check for AGENTS.md ... using `glob` or `readFile`." Whether the model actually acts on that is a per-turn judgment call, competing for attention against 7 other system-prompt sections ([system-prompt.ts:45-76](../../packages/server/src/prompts/system-prompt.ts#L45)) plus tool-usage rules. For short or conversational turns the model reasonably doesn't treat it as a "task" worth a tool round-trip, so the file's contents never enter context at all.

The server is architecturally filesystem-blind by design (`ai-workflow-rules.md`: "`@koincode/server` — AI orchestration only; no filesystem access, no tool execution"). All filesystem reads happen client-side in the CLI, in `packages/cli/src/tools/*` (`read-file.ts`, `write-file.ts`, `edit-file.ts`, `glob.ts`), dispatched through `executeLocalTool` in `packages/cli/src/tools/index.ts`.

Two existing patterns are directly relevant precedent:

1. **`userMemory`** ([system-prompt.ts:338](../../packages/server/src/prompts/system-prompt.ts#L338), `getMemorySection`) — actual content (pulled from the `Memory` table **fresh on every single `/chat` call**, `chat.ts:111-115`, not captured once at session start) is inlined into the system prompt every request. Not an instruction to go fetch memories — the memories are just there. Caching still works fine here in the steady state, because Anthropic's cache is a byte-exact prefix match, not a change-detection flag: identical `system` text hits the cache regardless of whether it was freshly reconstructed or reused. Only the specific turn where a memory actually changes pays a one-time cache-miss cost; caching resumes normally the next turn.
2. **`roots`** — captured once from the CLI at session creation (`POST /sessions`, `packages/server/src/routes/sessions.ts:155-178`) or via an add-dir-style update (`sessions.ts:190-210`), stored on the `Session` DB row (`Session.roots`, JSON), and parsed fresh (`parseWorkspaceRoots`) on every `/chat` request (`chat.ts:277`) to feed `buildSystemPrompt`. Persisted rather than re-derived per-request because there's no cheap client-side recomputation of "what are this session's workspace roots" — it's stateful, user-driven (`/add-dir`), not something read fresh off disk each time.

Also relevant: `packages/server/src/lib/prompt-caching.ts` establishes that `system` must stay **byte-stable across turns within a session** for caching to hold — Anthropic's cache is a cumulative prefix over `tools → system → messages`. `ideActiveFile`/`ideSelection` are deliberately appended onto the latest message instead ([prompt-caching.ts:20-33](../../packages/server/src/lib/prompt-caching.ts#L20)), but *not* because putting per-turn content in `system` would "break" caching in some absolute sense — it's because that content is expected to differ on nearly every turn (the user moves around their editor constantly), so keeping it in `system` would mean paying full reprocessing on nearly every turn. Attaching it to the last message instead is free, because per `prompt-caching.ts:27-29` that message is "already a fresh, uncached write every turn" regardless — a new user turn is new content no matter what. The two placement options aren't "cache-safe" vs. "cache-breaking" in general; they're a tradeoff that depends on how often the content actually changes.

## Goal

Global and root-level instruction file content (`AGENTS.md`, or `CLAUDE.md`/`CONTEXT.md` as fallbacks) is always present in the model's context, without requiring the model to decide to fetch it. Nested (per-subdirectory) instruction files are surfaced automatically the moment the model actually touches a file in their scope, without needing to preload every scoped file up front.

## Prior art: how opencode actually does this

Read directly from `anomalyco/opencode` (`packages/opencode/src/session/instruction.ts`, `system.ts`, `prompt.ts`, `tool/read.ts`) rather than inferred from behavior. Confirms the two-tier design below independently, and corrects a few details relative to the first draft of this spec:

- **Root/eager tier** (`instruction.ts`'s `systemPaths()`/`system()`): walks from the session's working directory up to the worktree root, collecting one instruction file per directory level — first name found among `AGENTS.md` → `CLAUDE.md` → `CONTEXT.md` (deprecated) wins for that level — plus a single global fallback (`~/.config/opencode/AGENTS.md`, or `~/.claude/CLAUDE.md` if the former is absent). `instruction.system()` is called fresh inside `Effect.all([...])` on **every agentic step** ([prompt.ts:1252-1262](https://github.com/anomalyco/opencode/blob/dev/packages/opencode/src/session/prompt.ts#L1252)), not cached or captured once — direct confirmation that the "read fresh every request, let the provider's cache diff itself" design (the correction earlier in this doc) is exactly what a shipping competitor does, not a theoretical concern.
- **Nested/lazy tier** (`instruction.ts`'s `resolve()`, wired in from `tool/read.ts:300,355-356` only): triggered by the `read` tool alone — not write/edit/glob. Walks from the read file's directory up to (not including) the project root, attaching each level's nearest instruction file if it isn't already covered by the eager root set and hasn't already been surfaced. The content is appended directly into that tool call's own output text, wrapped in `<system-reminder>...</system-reminder>`, rather than a separate field requiring bespoke UI handling: `output += "\n\n<system-reminder>\n" + loaded.map(i => i.content).join("\n\n") + "\n</system-reminder>"`.
- **"Already surfaced" tracking has no separate in-memory session map.** opencode stores `metadata.loaded: string[]` (the paths attached) directly on the persisted tool-call result, and re-derives "what's already surfaced" by scanning that field back out of prior messages in the conversation (`extract()` in `instruction.ts`) on each call. A small per-current-message `Set` (`claims: Map<messageID, Set<path>>`) additionally prevents attaching the same file twice within one burst of tool calls in a single step, but the durable, cross-turn dedup is entirely derived from message history — nothing tracked outside the conversation itself.
- **Compaction interaction is handled explicitly**: `extract()` skips any tool-result part where `time.compacted` is set, so a file's "already loaded" claim resets once the turn that first loaded it gets summarized away — otherwise the model would believe it still has instructions it actually lost.

## Scope decision: eager tier — global + per-root, fallback filename chain, read fresh on every request

Resolved (previously open questions): adopt opencode's `AGENTS.md` → `CLAUDE.md` → `CONTEXT.md` (deprecated) fallback chain, first name found wins, and add a global fallback file. Both decisions below.

**Filename chain**: for any directory being checked (each configured root's top level, and the global config dir), try `AGENTS.md` first; if absent, `CLAUDE.md`; if absent, `CONTEXT.md`. Included for exact parity with opencode, and because plenty of KOINCODE users will already have a `CLAUDE.md` from using Claude Code on the same project — requiring them to duplicate it as `AGENTS.md` just for KOINCODE to see it would be a needless bit of friction. First match wins per directory; the other names aren't also included if the winner is found (matches opencode's `find()`, one file per location, not a merge of all three).

**Global file**: `~/.koincode/AGENTS.md`, falling back to `~/.claude/CLAUDE.md` if the former doesn't exist — mirrors opencode's own `~/.config/opencode/AGENTS.md` → `~/.claude/CLAUDE.md` choice exactly, including piggybacking on Claude Code's file as the fallback rather than inventing a KOINCODE-specific path for it. `~/.koincode/` is already this codebase's established config directory (`~/.koincode/config.json` for provider keys, per `global-config.ts`), so `~/.koincode/AGENTS.md` is the natural fit, not a new convention. Exactly one global file is used (first found), same as opencode's `break` after the first match — no merging global `AGENTS.md` and global `CLAUDE.md` together if both happen to exist.

**Per-root files**: each configured workspace root (`roots`, not just the primary one) gets its own top-level check using the same three-name chain. Unlike opencode, no walk from cwd up to a worktree boundary — KOINCODE's `roots` aren't necessarily nested inside one shared worktree (they can be unrelated directories added via `/add-dir`), so v1 checks each root's own top level only, not intermediate ancestor directories between some cwd and a root.

**Mechanics**: same as the original per-request design — the CLI reads all of these fresh (global + each root) immediately before each `/chat` POST (a handful of local disk reads, microseconds total, negligible next to the LLM call) and includes them in that request's body, the same per-request mechanism already used for `ideActiveFile`. The server passes the bundle straight into `buildSystemPrompt` for that turn — no persistence anywhere. This doesn't cost caching anything in the steady state: unchanged files → identical bytes → the `tools → system` prefix still hits cache; a mid-session edit takes effect on the very next turn at the cost of one cache-miss turn, then caching resumes.

It's placed in `system` (unlike `ideActiveFile`) specifically *because* this content changes rarely — that's where it gets the most value from caching. Appending it to the per-turn message slot instead would mean paying its full token cost on every single turn with zero caching benefit.

Each entry is rendered with a source label, mirroring the `[global]`/`[project]` tagging convention already established in Feature 46 (`/mcp`'s status dialog) for the same global-vs-project merge problem: `Instructions from: <global | root label>` per block (format matches opencode's own `Instructions from: ${path}` exactly), replacing the current "go check for it" instruction at `system-prompt.ts:50`, not sitting alongside it.

## Scope decision: nested AGENTS.md is surfaced only via the `readFile` tool, tracked through message history — not a new in-memory map

Do not eagerly read every nested `AGENTS.md` in the project at session start (cache-bloat/dilution reasoning from earlier in this doc still holds, and this is now independently confirmed as opencode's own design, not just a KOINCODE-specific guess).

**Trigger**: `runReadFile` only, matching opencode's scope exactly — not `writeFile`/`editFile`/`glob`. `write-file.ts`/`edit-file.ts` do their own internal `readFile` calls to compute diffs, but that's not a model-issued `readFile` tool call, so a net-new file created without ever reading anything nearby first won't surface a nested `AGENTS.md` under this design. This is an accepted gap, not an oversight — opencode has the identical gap, and closing it would mean instrumenting write/edit too for a case (writing into a fresh subdirectory with no prior read) that's uncommon enough not to be worth the extra surface area for v1.

**Mechanism**: `runReadFile` resolves its target path (`resolveFromCwd`, `read-file.ts:45`) as it already does, then walks from that directory up to (not including) the nearest workspace root, checking the same `AGENTS.md` → `CLAUDE.md` → `CONTEXT.md` fallback chain from the eager tier at each level, collecting the nearest match not already covered by the eager root-level section above. For each one found, it checks the in-scope message list (already available to the CLI where `executeLocalTool` runs, in `use-chat.ts`) for a prior tool result carrying that same path in a new `loadedAgentsMd: { path, content }[]` field, and skips it only if the *content* matches too (see the third correction below — this replaced an earlier path-only version). New/changed ones get attached and recorded in this call's own `loadedAgentsMd`.

**No separate `Map<sessionId, Set<path>>`.** The first draft of this spec proposed exactly that; opencode's actual design doesn't need one, because the durable "already surfaced" state lives in the message history itself (`loadedAgentsMd` persists as part of the tool-call result in the message JSON blob, same as every other field on that return value already does).

**Correction found during implementation, not just planning, then fixed**: the second draft of this spec claimed this was "automatically compaction-safe for free" because `getLastBoundaryIndex` drops pre-boundary messages wholesale. That's true of the *server's* history-building path (`chat.ts`), but the client-side scan (`extractLoadedAgentsMdPaths`, reading `chat.messages` from the `useChat` hook) is a different array with a different lifecycle — verified against the actual code rather than assumed. `session.tsx:131-139` strips the `clear_boundary`/`compact_boundary` marker *objects* out of `initialMessages` before they ever reach the hook, and `handleClearSession`/`runCompact` only touch a separate render-only offset (`localClearMsgCount` — and `runCompact` doesn't even touch that one, by design: the transcript intentionally still shows pre-compact messages until the user leaves and returns, while `/clear` hides them immediately) — `chat.messages` itself is never truncated within a live process. Leaving the AGENTS.md tracking to inherit that same lag would be a real correctness bug, not a cosmetic one: the server has already dropped that history from what the model can see the moment a boundary is created, so treating a pre-boundary path as "still loaded" means the model silently loses access to it with no way to get it back short of a new session — worse than not deduping at all.

Fixed with a boundary counter dedicated to this feature, decoupled from `localClearMsgCount`: a new module-level `_lastInstructionBoundary: Map<sessionId, number>` in `use-chat.ts`, alongside the existing `_activeRoots`/`_activeModes` maps. `useChat` returns a new `markInstructionBoundary()` that records `chat.messages.length` at the moment it's called; `session.tsx`'s `handleClearSession` and `runCompact` both call it right after their respective POST succeeds (previously `runCompact` updated no client-side state at all for this purpose — the transcript's own lag is intentional and untouched, this is a separate, immediately-applied signal). The `readFile` call site slices `chat.messages.slice(_lastInstructionBoundary.get(sessionId) ?? 0)` before handing it to `extractLoadedAgentsMdPaths`, so both boundary types are immediately reflected in the same live process, no reload required.

**Second correction, same session — the live-only fix didn't survive a reload.** `_lastInstructionBoundary` is in-memory and gets deleted on unmount (same cleanup effect as `_activeRoots`), so reopening a session (navigate away and back, or a full CLI restart) lost any boundary that existed before that mount — right back to scanning full unsliced history for a session that had already compacted in a prior run. Fixed by reusing a value `session.tsx` already computes for an unrelated purpose: `countMessagesBeforeLastBoundary(rawSessionMessages)` (session.tsx:102-116), which finds the last boundary of *either* type in the raw DB payload (markers still intact) and returns an index into the marker-stripped coordinate space — currently used only to seed `localClearMsgCount`'s initial state. `useChat` now takes a 5th param, `initialInstructionBoundary`, and `session.tsx` passes its own `localClearMsgCount` (read at the moment of the call, i.e. still its mount-time value) straight through. `use-chat.ts` seeds `_lastInstructionBoundary` from it once, guarded by `!_lastInstructionBoundary.has(sessionId)` so it never clobbers a later live `markInstructionBoundary()` call on a subsequent render. The eager tier (global + per-root) was never affected by any of this — it re-reads fresh every turn regardless.

**Third correction, same session — dedup was path-only, so an edit to an already-surfaced file was never picked up.** User asked directly, separately from the boundary questions above: if a nested `AGENTS.md` has already been shown once and the user then edits it, does anything ever re-surface the new content? It didn't — `alreadyLoaded` was a `Set<string>` of paths; once a path was in it, it stayed excluded regardless of whether the file's actual content had changed since. (opencode has the identical gap — its `already.has(found)` check in `instruction.ts` is also purely path-based — so this wasn't a mistake unique to this implementation, but not a reason to leave it either.) Fixed by switching the whole chain from path-keyed to content-keyed: `runReadFile`'s `loadedAgentsMd` field is now `{ path, content }[]` instead of `string[]`; `extractLoadedAgentsMdPaths` was renamed `extractLoadedAgentsMd` and now returns `Map<path, content>` instead of `Set<path>`; `findUnsurfacedAgentsMd`'s `alreadyLoaded` param is the same map, and its skip condition changed from `alreadyLoaded.has(found.path)` to `alreadyLoaded.get(found.path) !== found.content` — a changed file is treated exactly like a new one and gets re-attached on the next read that touches it; an unchanged one still gets deduped, so the original point of tracking this at all (don't repeat identical content every read) is preserved.

**Fourth correction, same session — reading an instruction file directly duplicated its own content onto itself.** `findUnsurfacedAgentsMd`'s walk starts at `dirname(resolvedFilePath)` — the directory *containing* the file being read. If that file *is itself* an `AGENTS.md`/`CLAUDE.md`/`CONTEXT.md` (e.g. the model calls `readFile("sub/AGENTS.md")` directly), the first iteration's `findInstructionFile(current)` finds that same file sitting in its own directory and — nothing in the original code distinguished it from a genuinely different nested file — re-attached its content back onto itself as a redundant `<system-reminder>` block, verbatim, right after the identical primary content. Confirmed with a real read before fixing: `content` came back as `"<file text>\n\n<system-reminder>\nInstructions from: <same path>\n<same file text>\n</system-reminder>"`. This exact case was already in this spec's own "Prior art" research — opencode's `resolve()` in `instruction.ts` has `found === target` as an explicit skip condition — but that guard didn't make it into the initial port. Fixed by adding `found.path !== resolvedFilePath` to `findUnsurfacedAgentsMd`'s push condition, alongside the content-comparison check from the correction above.

**Presentation**: append the found content directly into `runReadFile`'s own returned `content`/output (wrapped distinctly, e.g. a trailing `\n\n<system-reminder>\nInstructions from: <path>\n<content>\n</system-reminder>` block), not a bolted-on field that the tool-result UI needs special-case rendering for. Matches opencode's `read.ts:355-356` exactly and avoids net-new work item 8 from the first draft entirely.

The root-scoped `AGENTS.md` (per-root, from the eager tier) is excluded from this walk — it's already in `system`, don't double-surface it as a tool result too.

## Net-new work required

1. **CLI: `findInstructionFile(dir)` helper.** New function (`packages/cli/src/tools/utils.ts`, sibling to `resolveFromCwd`/`formatWorkspacePath`): given a directory, tries `AGENTS.md` → `CLAUDE.md` → `CONTEXT.md` in order, returns the first that exists. Shared by both the eager tier (step 2) and the nested tier (step 5).
2. **CLI: read the eager tier fresh, per request.** Wherever `ideActiveFile` is gathered before a `/chat` POST (`packages/cli/src/hooks/use-chat.ts`): one global check (`findInstructionFile(~/.koincode)`, falling back to `~/.claude/CLAUDE.md` specifically if nothing found there — not the generic chain, since the global tier's fallback is a single named file, not another chain) plus one `findInstructionFile(root.path)` per configured root, swallowing read errors. Include as `instructionFiles?: { source: "global" | { root: string }; path: string; content: string }[]` in the request body, tagged per Feature 46's `[global]`/`[project]` convention.
3. **Shared: extend the chat request schema.** `chat.ts:46-75`'s `submitSchema` gains the field above, alongside the existing `ideActiveFile`/`ideSelection` fields.
4. **Server: thread it into the system prompt.** `chat.ts:96` destructures `instructionFiles` from the validated body and passes it to `buildSystemPrompt` at `chat.ts:278`. `system-prompt.ts`'s `SystemPromptParams` gains the same field; replace the call to `getAgentsMdSection()` at `system-prompt.ts:50` with a content-bearing version that renders `Instructions from: <label>` blocks (global first, then each root) when present, and falls back to something much shorter than today's instruction block when none exist (no file → no section, rather than prose about a file that doesn't exist).
5. **CLI: `findUnsurfacedAgentsMd(resolvedFilePath, roots, alreadyLoaded)` helper.** `packages/cli/src/tools/utils.ts`: walks from the file's directory up to (not including) its nearest workspace root, using `findInstructionFile` at each level, skipping a match only when `alreadyLoaded.get(path) === content` (content-keyed — see the third correction above; not just path presence).
6. **CLI: `extractLoadedAgentsMd(messages)` helper.** `packages/cli/src/lib/instruction-files.ts`: scans `tool-readFile` parts with `state === "output-available"` for a prior `loadedAgentsMd` output field, returning `Map<path, content>` (later messages overwrite earlier ones for the same path — last-shown content wins).
7. **CLI: wire both into `runReadFile`.** `read-file.ts` gains a third `alreadyLoadedAgentsMd: Map<string, string>` parameter, calls `findUnsurfacedAgentsMd` after resolving its target path, appends any results into its own returned `content` (trailing `<system-reminder>` block), and records what it attached in a new `loadedAgentsMd: { path, content }[]` field on its return (always present, possibly empty).
8. **CLI: thread it through.** `executeLocalTool` (`tools/index.ts`) gains the same trailing parameter, passed only to the `readFile` case. `use-chat.ts`'s call site computes it as `toolCall.toolName === "readFile" ? extractLoadedAgentsMd(chat.messages.slice(_lastInstructionBoundary.get(sessionId) ?? 0)) : undefined` — the scan only runs for the tool that needs it, sliced to the current live boundary.
9. **CLI: `_lastInstructionBoundary` map + `markInstructionBoundary()`.** `use-chat.ts`: a new module-level `Map<sessionId, number>` alongside `_activeRoots`/`_activeModes`, cleared on session unmount the same way. `useChat`'s return value gains `markInstructionBoundary()`, which records `chat.messages.length` when called.
10. **CLI: call it from both boundary actions.** `session.tsx`'s `handleClearSession` and `runCompact` each call `markInstructionBoundary()` right after their POST succeeds — deliberately separate from `localClearMsgCount` (the transcript's own render offset, which intentionally lags after a compact and isn't touched by this).

No `@koincode/database` schema change and no `POST /sessions` payload change — root `AGENTS.md` is transient per-request input, not session state. No general-purpose in-memory session map — the one map this feature does add is scoped narrowly to "index of the last live boundary," not a duplicate of the message-history-derived state.

## Explicitly out of scope for v1

- **No dedup/precedence logic beyond "nested overrides parent on conflict"** — the existing prose rule ("more-deeply-nested AGENTS.md files take precedence... when instructions conflict") is left as guidance text alongside the injected content, not enforced programmatically; conflict resolution stays the model's job.
- **No `writeFile`/`editFile`/`glob`-triggered surfacing** — `readFile` only, matching opencode's own scope (see above). A net-new file written without a prior read won't trigger nested-`AGENTS.md` discovery.
- **No merging of multiple matches at the same location** — if a directory has both `AGENTS.md` and `CLAUDE.md`, only the higher-priority one is used, never both. Same as opencode.
- **No walk from cwd up to a worktree boundary for the eager tier** — each configured root's own top level only (see Scope decision above); not a factor for KOINCODE's actual usage pattern today (roots are added explicitly via `/add-dir`, not implied by cwd nesting), but worth revisiting if that changes.

## Package boundaries

Touches three packages, not four: `@koincode/shared` (`submitSchema`'s new field), `@koincode/server` (`chat.ts`, `system-prompt.ts`), `@koincode/cli` (`use-chat.ts`, `tools/index.ts`, `tools/utils.ts`, `read-file.ts`, `lib/instruction-files.ts`, `screens/session.tsx`). No `@koincode/database` change. Per `ai-workflow-rules.md`'s splitting rule ("Do not combine changes across unrelated packages in a single implementation step"), this landed as two implementation steps: (a) root `AGENTS.md` — schema field → per-request read → system prompt; (b) nested `AGENTS.md` — the `readFile`-only tool-layer hook plus the live-boundary fix that followed once its message-history-scan approach was checked against the actual client architecture rather than assumed correct.

## Open questions

None outstanding — resolved above: adopt the `AGENTS.md` → `CLAUDE.md` → `CONTEXT.md` fallback chain (first found wins per location); add a global fallback at `~/.koincode/AGENTS.md`, itself falling back to `~/.claude/CLAUDE.md`.
