# Feature 36: Prompt Caching Support

## Origin

Follow-on from discussing subscription-based auth (`xx-external-provider-support.md`): since chat APIs are stateless, every turn re-sends the entire system prompt, tool definitions, and prior conversation history as input tokens — cost compounds as a session grows. Several providers offer prompt/context caching to avoid re-billing (and often re-processing) identical prefixes across requests. This spec covers what KOINCODE can actually do about it, per provider.

## Goal

Cache the stable, repeated parts of every request (system prompt, tool definitions, and — ideally — the growing conversation-history prefix) wherever the provider supports it, so a long session's *effective* cost per turn stops scaling with the full accumulated context and instead scales with just the new tokens added that turn.

## Provider-by-provider reality (checked against the installed SDK and this repo's code, not assumed)

- **Anthropic — in scope, real work.** Caching is opt-in via an explicit `cache_control` marker. Confirmed the installed `@ai-sdk/anthropic@3.0.68` exposes this as `providerOptions.anthropic.cacheControl: { type: "ephemeral", ttl?: "5m" | "1h" }`, attached per message/content-part (not a single top-level request flag). Cached reads bill at ~10% of normal input price; a cache write costs a premium (~1.25× for 5m TTL) the first time a prefix is cached. Nothing in the codebase sets this today (grepped — `providerOptions` is currently only used for thinking-mode config and Ollama's `num_ctx`).

- **OpenAI — no new caching code, but a prerequisite fix applies here too.** OpenAI's caching is automatic and transparent (repeated prefixes ≥1024 tokens get cached server-side, no request changes needed) — *provided* the prompt actually stays byte-identical turn to turn up to some point. That precondition is currently broken by the same system-prompt-ordering issue described below, which affects both providers.

- **Google (Gemini) — out of v1, heavier lift.** Context caching on Gemini is an explicit lifecycle API: you create a cached-content object with its own TTL and reference it by ID in subsequent requests, rather than an inline per-message marker. That's a meaningfully bigger integration (cache creation, expiry/refresh, cleanup) than Anthropic's inline breakpoints. Flagged as a follow-up, not v1.

- **OpenRouter — in scope, done.** Initially written off as "unresolved" — wrong, corrected after the user pushed back and it was checked against OpenRouter's own docs plus the installed `@openrouter/ai-sdk-provider@2.9.0` type declarations. Two mechanisms, both confirmed real: (1) OpenAI models routed through OpenRouter cache automatically with zero config, same as calling OpenAI directly; (2) Anthropic models get a top-level `cache_control: { type: "ephemeral", ttl? }` setting (confirmed field on `OpenRouterChatSettings`, passed once at model-construction time via `openrouter.chat(modelId, settings)`) that OpenRouter auto-advances as the conversation grows — no manual per-turn breakpoint bookkeeping needed on our side, unlike the direct-Anthropic-SDK path. One caveat from OpenRouter's docs: automatic caching only engages when OpenRouter's routing actually lands on Anthropic's own backend, not a Bedrock/Vertex proxy of the same model — not something worth over-engineering around for v1.

- **Custom OpenAI-compatible providers / Ollama — out of scope.** There's no universal caching mechanism across arbitrary self-hosted or third-party OpenAI-compatible backends (`createOpenAICompatible` in `resolveCustomModel`) — support would vary per backend with no consistent API to target. Ollama isn't billed per token at all, so the cost motivation doesn't apply there; any prefix/KV-cache reuse Ollama or vLLM does internally happens at the inference-server level, outside KOINCODE's control and already free.

## Prerequisite fix: system prompt ordering (blocks both Anthropic and OpenAI caching)

Read `packages/server/src/prompts/system-prompt.ts` directly to verify this rather than going on a paraphrase:

- `buildSystemPrompt` (line 34-72) pushes sections in this order: identity → environment → AGENTS.md → mode → tool usage → security → coding guidelines → operational → (conditionally) browser control → visualization → skills → **`ideActiveFile`** (line 58-63) → userMemory → compact summary.
- Everything before `ideActiveFile` is either fully static or stable for the life of a session (mode, tool list, skills manifest, MCP server list — these only change on the rare event of a skill being added or an MCP server reconnecting mid-session).
- `getEnvironmentSection()` (line 87-108) interpolates `new Date()` — but formatted with `toLocaleDateString("en-US", { weekday: "long", year: "numeric", month: "long", day: "numeric" })`, i.e. **day granularity, no time-of-day component**. Verified directly: this does not actually change within a cache's 5-minute/1-hour TTL window under normal operation, so despite being recomputed every call, it is *not* a practical cache-buster in itself — no fix needed here.
- **`ideActiveFile` is the real problem.** It reflects whichever file is open in the user's editor and can change turn-to-turn as they switch tabs (per `packages/cli/src/hooks/use-ide-context.ts`'s file-watching). It currently sits in the *middle* of the prompt (before `userMemory` and the compact-summary section), so any cache breakpoint placed after it would constantly miss, and — for OpenAI's automatic caching — any content placed after this point in the string never benefits from caching either, since automatic caching also requires an unchanged prefix.

**Fix:** move the `ideActiveFile` block to be pushed last in `buildSystemPrompt` (after `userMemory` and `getCompactSummarySection()`), so the entire stable portion of the prompt forms one unbroken prefix, with only the genuinely per-turn-volatile content trailing at the very end. Mechanical, low-risk, and benefits OpenAI's automatic caching immediately with zero other changes.

**⚠️ Superseded — see "Second correction" in Status.** This reorder-within-the-system-prompt approach turned out to be the wrong fix for Anthropic specifically: caching there is a cumulative prefix across `tools → system → messages` in that canonical order (confirmed against Anthropic's docs), so *any* per-turn-varying content anywhere in `system` — reordered or not — sits upstream of the `messages` breakpoint and spoils it. The actual fix moves `ideActiveFile` out of `system` entirely, onto the newest message. Left here for the historical record of how the diagnosis evolved.

## Design (Anthropic, v1)

1. **System prompt + tools as one cached block.** Attach a `cacheControl: { type: "ephemeral" }` marker so the system content and tool definitions are cached together as the first, largest, most stable chunk. Exact mechanism needs verification at implementation time: `chat.ts` currently passes `system` as a plain string convenience field to `streamText` (`packages/server/src/routes/chat.ts:239-246`) — need to confirm whether the installed `ai@^6` version lets you attach per-message `providerOptions` to that shorthand, or whether the system prompt needs to move into an explicit system message object in the `messages` array to carry the marker. Same applies to whether a tool definition can carry its own `providerOptions.anthropic.cacheControl` to close out the tools block. Verify against the actual installed type declarations before writing code — same practice the project already followed when it hit the `InputRenderable`/`TextareaRenderable` type-shape surprise in Feature 34 Phase 2.
2. **Conversation-history breakpoint.** A `cacheControl` marker on the newest message, so each turn reuses the cache built by the previous turn instead of re-billing the entire growing history at full price — this is the piece that actually addresses the cost-compounding problem, not just the fixed system-prompt overhead. Must move forward each turn. Interacts cleanly with the existing "always slice from last boundary" rule (`getLastBoundaryIndex`) — post-compact, the boundary reset naturally invalidates the old cache and a fresh one builds from the new post-compact context, which is expected and correct, not a bug to work around. **This is also where the per-turn volatile `ideActiveFile` context now gets attached** (see Status) — it has to live on this same newest message, not in `system`, because Anthropic's cumulative canonical order (`tools → system → messages`) means anything volatile placed in `system` would spoil this exact breakpoint regardless of where in the system prompt it sits.
3. **Gating.** Both markers only apply when `resolvedModel.provider === "anthropic"` and the direct Anthropic key path is in use (mirrors how `ANTHROPIC_THINKING` providerOptions are already conditionally attached in `packages/server/src/lib/models.ts`).

## Net-new work required

1. **System prompt reorder** — move `ideActiveFile` to be pushed last in `buildSystemPrompt`. No schema/type changes. Benefits OpenAI immediately, unblocks Anthropic caching from being undermined by the same issue.
2. **Verify AI SDK attachment points** for `cacheControl` against the installed `@ai-sdk/anthropic@3.0.68` types — confirm whether `system` needs to move from a plain string into an explicit system message object, and whether tool definitions can carry their own marker. This is a research/verification step, not something to guess at in code.
3. **Anthropic system+tools cache breakpoint** — implement based on what step 2 confirms.
4. **Anthropic history cache breakpoint** — advance-per-turn marker on the sliced message array.
5. **Cost estimator update** (`lib/cost.ts`'s `estimateSessionCost`, used by the info sidebar and `/context`) — once caching is live, Anthropic's usage response includes separate `cache_creation_input_tokens` / `cache_read_input_tokens` fields billed at different multiples of the base input rate (~1.25× write, ~0.1× read) rather than the flat `inputUsdPerMillionTokens` rate `ModelPricing` currently assumes. Without this, the cost shown in the info sidebar would become actively wrong (understated or overstated) as soon as caching ships — this is required to keep an existing feature correct, not scope creep. Can likely be derived as multiples of the existing per-model input rate rather than adding new pricing fields to every model entry.
6. **OpenAI verification pass** — after step 1 ships, confirm (via real usage data / OpenAI's cached-token usage field) that automatic caching is actually landing on KOINCODE requests. No new code expected here, just confirmation.

## Explicitly out of v1 (open questions, not guesses)

- **Google context caching** — separate cache-lifecycle API, real integration work; revisit as its own feature spec if it's worth the lift once Anthropic ships.
- **Custom/Ollama providers** — no universal mechanism, no per-token billing motivation for Ollama specifically.

## Package boundaries

Server-only feature (`packages/server/src/lib/models.ts`, `packages/server/src/routes/chat.ts`, `packages/server/src/prompts/system-prompt.ts`) plus one CLI-side change (`packages/cli/src/lib/cost.ts`'s cost estimator, to stay correct once cache-priced tokens start showing up in usage data). No schema changes, no new tool contracts.

## Status

**Items 1-5 done, corrected once by a second pass.** Item 6 still needs a live account, noted below.

- **First correction (superseded by the second, below):** the initial reorder (moving `ideActiveFile` to the end of the joined system-prompt string) was necessary but not sufficient for Anthropic — a single content block invalidates as a whole on any change regardless of position. Fixed by splitting system into `{ stable, volatile }` as two separate `SystemModelMessage`s, cache marker only on `stable`.
- **Second correction (the actual final design) — confirmed directly against Anthropic's docs, not assumed:** Anthropic's cache is a *cumulative* prefix in the fixed canonical order `tools → system → messages`, regardless of request field order. A breakpoint's hit depends on **everything before it in that order**, not just its own block. That meant even with `ideActiveFile` isolated in its own uncached `system` block, it still sat upstream of the `messages` breakpoint in the canonical order — so it silently spoiled the *history* cache breakpoint (the one that actually matters for stopping cost from compounding) every time the user switched editor tabs, while leaving the `tools` breakpoint (canonically first, unaffected by anything in `system`) untouched. Caught by the user asking "doesn't the volatile spoil the whole thing?" before this ever shipped to a real request.
  - **Real fix:** `ideActiveFile` no longer lives in the system prompt at all. `buildSystemPrompt` (`packages/server/src/prompts/system-prompt.ts`) reverted to a single fully-stable string (the `{ stable, volatile }` split and `ideActiveFile` param removed entirely — no `SystemPromptParts` export anymore).
  - New `appendIdeContext(messages, ideActiveFile)` in `packages/server/src/lib/prompt-caching.ts` appends the IDE-context text directly onto the **newest message's own content** instead — that message is already a fresh, uncached write every turn, so attaching volatile text there costs nothing extra, and once finalized it never changes again so it can't threaten *future* turns' cache hits either. Skipped when the last message is a tool-result continuation (`role: "tool"`) — that content type has no text-part slot, and there's no fresh user prompt to attach live context to in that request anyway (an accepted, minor trade-off, not a bug).
  - `buildCachedSystemMessages` → simplified to `buildCachedSystemMessage(systemPrompt, enableCaching)`, returning one `SystemModelMessage` (system is one block again, no volatile counterpart to carry).
  - Ordering in `chat.ts` matters: `appendIdeContext` runs *before* `withHistoryCacheControl`, so the cache breakpoint's hash covers the final, IDE-context-included content of the newest message, not a stale pre-append snapshot.
- **`ResolvedModel.promptCaching`** (`packages/server/src/lib/models.ts`) — new boolean, set `true` only in `resolveAnthropicModel`'s direct-key branch. Explicitly *not* inferred from `provider === "anthropic"` alone, since that field stays `"anthropic"` even on the OpenRouter-fallback path — that path uses a *different*, separate mechanism (below), not `chat.ts`'s manual breakpoint helpers.
- **OpenRouter caching** (`packages/server/src/lib/models.ts`'s `resolveViaOpenRouter`) — self-contained, no `chat.ts` involvement at all. Passes `{ cache_control: { type: "ephemeral" } }` as the `settings` argument to `openrouter.chat(routerModelId, settings)`, gated on `provider === "anthropic"` (the only case in this codebase where `resolveViaOpenRouter` is known to be routing an Anthropic model — the genuine `"openrouter"`-native model list has no `anthropic/*` entries today, so this gate is exact, not a heuristic). OpenAI-via-OpenRouter needed no change (automatic on OpenRouter's end, same as calling OpenAI directly).
- **`packages/server/src/lib/prompt-caching.ts`** — final shape, all confirmed against the installed `@ai-sdk/anthropic@3.0.68` / `ai` types (`SystemModelMessage`, `ModelMessage`, and `Tool` all carry an optional `providerOptions` field; `UserContent`/`AssistantContent` allow a `TextPart`, `ToolContent` does not — confirmed by reading the type declarations directly, which is what surfaced the tool-continuation caveat above):
  - `buildCachedSystemMessage(systemPrompt, enableCaching)` — the (now fully stable) system prompt as one message, cache marker when enabled.
  - `appendIdeContext(messages, ideActiveFile)` — as described above.
  - `withToolsCacheControl(tools, enableCaching)` — marks the last tool definition, closing out the `tools` prefix as one cached unit (canonically first, so this one was never actually at risk from the `ideActiveFile` bug).
  - `withHistoryCacheControl(messages, enableCaching)` — marks the last message in the sliced array every turn, so each turn's history becomes the next turn's cache hit — now actually reliable, since nothing volatile sits upstream of it in the canonical order anymore.
- **`packages/server/src/routes/chat.ts`** — main `POST /` handler wires all of the above into the `streamText` call, gated on `resolvedModel.promptCaching === true`. `/agent-step` untouched — single-step ephemeral calls, no accumulating history to benefit from caching, out of scope for this pass.
- **`packages/cli/src/lib/cost.ts`** — `estimateSessionCost` now reads `usage.inputTokenDetails.{noCacheTokens,cacheReadTokens,cacheWriteTokens}` (confirmed present on the AI SDK's `LanguageModelUsage` type) and prices each at the correct rate (read ≈0.1× input rate, write ≈1.25×) instead of billing all input tokens at the flat rate. Falls back to the old flat-rate behavior byte-for-byte when a provider doesn't report the breakdown (non-Anthropic models, or historical turns saved before this shipped).

- **Item 6 (OpenAI verification) still open** — needs a real OpenAI account/usage response to confirm automatic caching is actually landing; nothing to check statically here.
- **Not done, by design:** only Google's context-caching lifecycle remains out of scope now — OpenRouter (originally written off as unresolved) is done, see above.
