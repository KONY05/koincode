i noticed that after quitting and restarting the CLI, a backgrounded shell command (`run_in_background: true`) shows its spinner forever instead of reflecting that it actually finished — traced that to `background-process-status.ts` being a purely in-memory, process-lifetime-only tracker, same accepted limitation as `scheduleWakeup`'s timer.

Digging into a proper fix (patching the persisted DB message row when the process exits) turned up a bigger problem underneath: the agent itself has no way to ever learn a backgrounded shell command's result — not just across a restart, but even within the same live session. `run_in_background: true` returns `{ pid, message }` once, at spawn time, and nothing else ever tells the model what happened. Compare to `spawnAgent`'s `runInBackground`, which has a full completion pipeline (`agent-background-tasks.ts`, default auto-delivery, `scheduleWakeup`'s `waitingOnTaskId`) — `shell` never got the equivalent.

Separately, while looking at how `spawnAgent`'s background delivery actually reaches the model: it's sent via `chat.sendMessage()` as a real `role: "user"` turn, so it renders in the transcript as an ordinary user-typed bubble ("Sub-agent \"X\" (task Y) finished...") — indistinguishable from something the human said. Want that to read as the assistant surfacing its own findings instead, while the model still gets a real turn to react to it.

## Related, already shipped (separate, smaller fix)

`shell.tsx` / `background-process-status.ts` already got a standalone fix in the same investigation: distinguishing "no tracked entry for this PID" (meaning it predates the current process — registration happens synchronously at spawn, so an untracked PID can only mean a previous run) from "tracked, not yet exited." The UI now renders `⏸ background (unknown — session restarted)` instead of a live-looking spinner in that case. This is purely a display-honesty fix for the human; it does **not** address the agent-visibility gap below, and this spec doesn't touch it further.

A DB-patch approach (find the message row by `toolCallId`, mutate the persisted tool-result JSON to add `exitCode`) was explored and rejected: it requires a schema migration, a new server endpoint, and optimistic-concurrency handling (SQLite has no row-level `SELECT ... FOR UPDATE`, so it'd need an `updatedAt`-based compare-and-swap retry loop) to avoid clobbering the same message row `chat.ts`'s own turn-save logic writes to. The delivery-as-a-new-turn approach below sidesteps that race entirely — nothing writes back into old history after the fact.

## Decision

Two related changes:

1. **Shell background result delivery parity with `spawnAgent`** — give `run_in_background` shell commands the same tracked-completion + default-auto-delivery + `scheduleWakeup`(`waitingOnTaskId`) pipeline `spawnAgent`'s `runInBackground` already has, so the agent actually learns stdout/stderr/exitCode once the process exits, instead of never finding out.
2. **Render background-delivered results on the assistant side.** Tag these delivery messages with a metadata flag so the transcript renders them through the assistant-side component instead of as a fake user-typed bubble — the underlying mechanism (a real new turn sent to the model) is unchanged; only how the CLI chooses to draw it changes. Applies to both the existing `spawnAgent` delivery and the new shell delivery from (1).

## Design

### Part A — Shell background result delivery

- Generalize the completion-registry pattern instead of duplicating it: `agent-background-tasks.ts` (`createAgentBackgroundTask`/`completeAgentBackgroundTask`/`failAgentBackgroundTask`/`getAgentBackgroundTask`/`onAgentTaskSettled`) is currently spawnAgent-specific by name and intent. Either generalize it to a shared registry both `spawnAgent` and `shell` register into, or add a parallel, identically-shaped registry for shell PIDs — so `scheduleWakeup`'s `waitingOnTaskId` and the default-delivery pattern work the same way for both without `scheduleWakeup` needing to know which kind of background work it's waiting on.
- `runShellCommand`'s `run_in_background` branch (`shell.ts`) registers a task the moment it spawns. The PID itself is a natural, already-unique id — no need for a separate `crypto.randomUUID()` scheme like `spawnAgent` uses, unless the generalized registry needs a single id type across both kinds of task (in which case stringify the PID).
- On `proc.exited`, in addition to the existing `markProcessExited(pid, exitCode)` (kept as-is for the UI's `onProcessExited` subscription), also settle the task registry entry with a formatted result — same style as `spawnAgent`'s delivery text: `Background shell command (PID <pid>, "<command>") finished — exit <code>.\n\nOutput:\n<stdout/stderr, truncated>`.
  - Truncation direction: keep the **tail**, drop from the **top** — the opposite of the tool-view's existing `stdoutLines.slice(0, MAX_LINES)` (`shell.tsx`), which keeps the head. For a delivered message the most recent output (final errors, completion status) is what the agent actually needs to act on, and that's almost always at the end. This is specific to the delivery message text; the tool-view's own head-truncated display for the live/foreground case is unrelated and unchanged.
- No dedicated `checkShellTask` poll tool — default delivery + `scheduleWakeup`'s optional `waitingOnTaskId` is sufficient, matching `checkAgentTask`'s optional (not required) role in the `spawnAgent` pattern.
- Default delivery: the moment the task is created, subscribe via the registry's `onSettled`-equivalent with a default listener that pushes the result onto the message queue unconditionally — mirroring the "Follow-up: default delivery" fix already applied to `spawnAgent` (see `13-subagent-tool-implementation.md`), so `scheduleWakeup` stays optional rather than required for the result to ever reach the agent.
- Update the `shell` tool's description (`schemas.ts`) to stop framing `run_in_background` as fire-and-forget — state that the result is delivered automatically, and that `scheduleWakeup` (`waitingOnTaskId`) is available for a specific timed check-in, mirroring `spawnAgent`'s current description.
- Still process-lifetime-only, same as every other in-memory registry in this codebase (`background-process-status.ts`, `scheduleWakeup`'s timer, `agent-background-tasks.ts` today) — not solved here, and not this spec's job to solve. If the CLI restarts while a backgrounded shell command is still running, the agent loses the ability to be notified, exactly like `scheduleWakeup`'s timer does today.

### Part B — Assistant-side rendering for background deliveries

- Add `origin?: "background-task"` to `ChatMessageMetadata` (`packages/shared/src/chat.ts`).
- `queueMessage` (`use-chat.ts`) needs a way to stamp this metadata — either an optional second param, or a thin wrapper used by all background-delivery call sites (the existing `spawnAgent` default listener, `scheduleWakeup`'s task-settle listener, and the new shell delivery from Part A) instead of calling `queueMessage` directly.
- `ChatMessage()` in `session.tsx`: when `msg.role === "user" && msg.metadata?.origin === "background-task"`, render via `<BotMessage parts={[{ type: "text", text }]} model={msg.metadata?.model ?? "unknown"} />` instead of `<UserMessage>`. `BotMessage` already renders a plain text part fine with no tool-view chrome, so no new rendering component needed.
- The wire-level role sent to the model stays `user` — required for the API/tool-result-continuation mechanics to work at all. Only the local render choice changes; nothing about how the model receives or processes the message changes.
- Whatever the model says *next* in reaction is a genuine, freshly-streamed assistant turn — needs no special handling, since it was never anything other than real model output.
- Renders fully indistinguishable from organic assistant text — no dim label, no "↩ background result" marker or similar. It should read exactly as if the assistant surfaced the finding itself.
- Double-check nothing else keys off `msg.role === "user"` in a way this breaks (`countMessagesBeforeLastBoundary`, message-queue draining, `/handoff`, compaction) — this only changes the render switch, not the role itself, so those should be unaffected, but worth a pass since they weren't written with this case in mind.

## Package boundaries

CLI + shared only, same reasoning as `40-wakeup-tool.md` — one schema field addition (`ChatMessageMetadata.origin`), everything else in the CLI package (`use-chat.ts`, `session.tsx`, `shell.ts`, `agent-background-tasks.ts` generalization). No server or database changes — this fully replaces the DB-patch direction explored earlier in favor of delivery-as-a-new-turn.

## Suggested implementation order

1. Generalize (or parallel) the background-task registry so both `spawnAgent` and `shell` settle through the same shape. Verifiable alone: `bun run typecheck` clean, existing `spawnAgent` background flow unaffected.
2. Wire `shell`'s `run_in_background` to register/settle through it, with default delivery on exit. Verifiable alone: background a shell command, do nothing else, confirm a follow-up message with the real output arrives without polling.
3. Update `shell`'s tool description + system prompt rules to match the new delivered-automatically framing.
4. Add the `origin` metadata flag and the `ChatMessage()` render switch. Apply first to the *existing* `spawnAgent`/`scheduleWakeup` deliveries — smallest verifiable slice, since it's a pure rendering change to something already working. Verifiable alone: background a sub-agent, confirm its result now renders on the assistant side instead of as a user bubble, and the model's next reply still follows naturally.
5. Apply the same flag to the new shell delivery from step 2.

## Open questions / deferred

All prior open questions resolved — see the corresponding Design bullets above (no dedicated poll tool, tail-first truncation for delivered output, fully indistinguishable assistant-side rendering).

- Cross-restart delivery is explicitly out of scope — same accepted process-lifetime ceiling as everything else here. The already-shipped "unknown — session restarted" UI fix covers the display-honesty half of that gap; there's no plan to deliver a stale result to the agent after a restart.

- **Registry generalized, not parallelled**: `agent-background-tasks.ts` was renamed back to `background-tasks.ts` (reversing the earlier agent-specific rename from `13-subagent-tool-implementation.md` — that rename's stated reason, "used for only agent background task," no longer holds now that shell registers into it too) and its types/functions genericized: `BackgroundTask`/`BackgroundTaskStatus`, `createBackgroundTask`/`completeBackgroundTask`/`failBackgroundTask`/`getBackgroundTask`/`onTaskSettled`. `createBackgroundTask(name, description, id?)` — `id` defaults to a generated uuid slice (spawnAgent's case, no natural id) or is passed explicitly as the stringified PID (shell's case). This means `scheduleWakeup`'s `waitingOnTaskId` and the default-delivery pattern in `use-chat.ts` work identically for both without ever needing to know which kind of task an id belongs to, exactly as the Design section called for.
- **Shell background wiring** (`shell.ts`): `run_in_background` now switches `stdout`/`stderr` from `"ignore"` to `"pipe"` (needed to capture output for delivery), registers a task at spawn (`createBackgroundTask("shell", description, String(proc.pid))`), and on `proc.exited` formats a delivery message (`Background shell command (PID <pid>, "<command>") finished — exit <code>.\n\nOutput:\n<tail-truncated stdout+stderr>`) via the new `truncateTail` helper (`utils.ts` — keeps the tail, drops from the top, opposite of `truncate`). A **non-zero exit code still calls `completeBackgroundTask`, not `failBackgroundTask`** — same reasoning as the earlier `ShellView` fix (a non-zero exit can be a normal, informative result, e.g. a `ps -p` liveness check); `failBackgroundTask` is reserved for the process actually throwing (caught via try/catch around the whole delivery pipeline). The tool's immediate return message and both `shell`'s tool-level and `run_in_background` field-level descriptions (`schemas.ts`) were updated to state delivery is automatic, mirroring `spawnAgent`'s phrasing.
- **Default delivery for shell** lives in `use-chat.ts`'s generic dispatch path (not a new `onToolCall` special case — that would have bypassed the permission gate and `PreToolUse`/`PostToolUse` hooks that regular shell calls must still go through, unlike `spawnAgent`/`scheduleWakeup` which intentionally skip them). After `executeLocalTool` resolves for `toolName === "shell"`, if the output shape is the background one (`{ pid, message }`, no `exitCode`), a default `onTaskSettled` listener is registered keyed by `String(pid)` — same shape and lifecycle as `spawnAgent`'s `_defaultTaskListeners` entries, and colliding id namespaces isn't a concern (agent ids are uuid-slice hex+dashes, shell ids are pure digit PIDs).
- **`scheduleWakeup`'s `waitingOnTaskId`** description (`schemas.ts`) updated to mention both task kinds; the system prompt's build-only rule 7 (`system-prompt.ts`) updated to describe automatic delivery for both backgrounded shell commands and `runInBackground` sub-agents, and to note `waitingOnTaskId` accepts either a `spawnAgent` `taskId` or a shell PID (as a string).

- **Bug found via code review: `timeout` was silently ignored for `run_in_background`.** The existing `shell` schema's `timeout` field (documented simply as "Timeout in milliseconds") was applied only in the foreground path (`setTimeout(() => proc.kill(), timeout)`) — the background branch never read it at all, so passing `timeout` alongside `run_in_background: true` silently no-op'd, contradicting the schema's own documentation. Fixed: `timeout` destructured without the JS default-parameter fallback it previously had; the foreground path now applies `timeout ?? DEFAULT_TIMEOUT` explicitly, while the background path only starts a kill timer if `timeout` was explicitly passed — no default, since backgrounded commands (builds/tests/watch processes) are expected to potentially run indefinitely by design, and forcing the same 30s default there would kill most of them prematurely. A background command that does get killed by an explicit timeout now delivers a distinct message (`"timed out after <ms>ms and was killed"` instead of `"finished — exit <code>"`) via a local `timedOut` flag set in the timer callback.
