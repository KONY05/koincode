### Improvement I
## Guide against unhandled shell command edge cases
previously didn't check the shell command patterns carefully and let user run potentially harmful commands.

**Change Breakdown**
# Permission System — What Changed & Why

## The Core Problem

The old `getShellPermissionInfo` did this:

```ts
const bin = command.trim().split(/\s+/)[0] ?? "";
const mapped = SHELL_BIN_MAP[bin];
```

It only ever looked at the **first word** of the command. So for any concatenated command, only the first sub-command counted. Everything after `&&`, `||`, `;`, or `|` was invisible to the permission system.

---

## 1. `splitSubcommands(command)` — Tokenise the full command

**What it does:** Walks the command string character by character and splits it on shell operators (`&&`, `||`, `;`, `|`), but **skips operators that appear inside quotes** so string arguments aren't accidentally split.

```ts
splitSubcommands('cd /tmp && rm -rf .')
// → ["cd /tmp", "rm -rf ."]

splitSubcommands('git commit -m "feat: add && fix"')
// → ["git commit -m \"feat: add && fix\""]   (the && inside quotes is ignored)

splitSubcommands('npm test; git push')
// → ["npm test", "git push"]
```

**Why it matters:** Without this, a command like `cd /tmp && rm -rf .` is a single opaque string. With this, we get individual pieces we can reason about.

---

## 2. `AtomicClassification` type + `classifyAtomicCommand(cmd)` — Classify one piece

**What it does:** Takes a single atomic command (no operators) and maps its leading binary to the `SHELL_BIN_MAP`. Returns a fully-typed object that always has `requiresApproval: true` plus an extra `isCdOnly` boolean flag.

```ts
classifyAtomicCommand("rm -rf .")
// → { key: "shell:rm", tier: "destructive", isCdOnly: false, ... }

classifyAtomicCommand("git status")
// → { key: "shell:git", tier: "normal", isCdOnly: false, ... }

classifyAtomicCommand("cd /tmp")
// → { key: "shell:cd", tier: "normal", isCdOnly: true, ... }

classifyAtomicCommand("some-unknown-cli --flag")
// → { key: "shell:unknown", tier: "normal", isCdOnly: false, ... }
```

The `isCdOnly` flag is the key insight: a bare `cd` is not a meaningful action on its own — it just sets the directory for what comes next. Marking it lets us skip it during the merge step.

**Why the explicit type matters:** The old code returned `PermissionInfo & { isCdOnly? }`, which includes the `{ requiresApproval: false }` branch from the union. That branch has no `.key` or `.tier`, so TypeScript complained whenever we accessed those fields. `AtomicClassification` pins the type to always-true, so the merge logic is clean with no casts.

---

## 3. `KEY_RANK` + `TIER_RANK` — Ordinal severity tables

```ts
const KEY_RANK = {
  "shell:cd":       0,  // harmless redirect
  "shell:git":      1,  // read/write repo
  "shell:npm":      1,  // package ops
  "shell:write":    2,  // filesystem writes
  "shell:rm":       3,  // deletion
  "shell:unknown":  4,  // anything we don't recognise — assume worst
  "file:sensitive": 5,
};
```

These numbers give us a way to compare classifications without a giant if/else chain. Higher = more dangerous.

---

## 4. New `getShellPermissionInfo(command)` — The merge step

**What it does:**
1. Calls `splitSubcommands` to get all the pieces.
2. Calls `classifyAtomicCommand` on each piece.
3. Filters out the `cd`-only pieces — unless *all* pieces are `cd`, in which case it keeps them (a lone `cd` is still a valid thing to classify).
4. Runs `reduce` over the remaining classifications, picking the one with the highest rank.
5. Returns a single `PermissionInfo` where the `key`/`tier` reflect the **most dangerous sub-command**, but the `description` is always the **full original command** (so the user sees exactly what they're approving).

```
Command                          Before               After
─────────────────────────────────────────────────────────────────
cd /tmp && rm -rf .              shell:cd (normal)  → shell:rm  (destructive) ✓
cd /project && git status        shell:cd (normal)  → shell:git (normal)      ✓
cd /project && unknown-tool      shell:cd (normal)  → shell:unknown (normal)  ✓
cd /project                      shell:cd (normal)  → shell:cd  (normal)      ✓ (unchanged)
git status                       shell:git (normal) → shell:git (normal)      ✓ (unchanged)
npm test; git push               shell:npm (normal) → shell:npm (normal)      ✓ (npm wins by rank tie, but either is correct)
```

The dangerous case is the first one. Before, `rm -rf .` hiding behind a `cd` would have been classified as a *normal* `shell:cd` and potentially auto-approved if the user had previously approved `shell:cd` for the project.
