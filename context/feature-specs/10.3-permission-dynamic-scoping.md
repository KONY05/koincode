# Feature Spec 10.3: Dynamic Permission Scoping for Shell Commands

## Problem Statement

The shell command permission system currently uses a static `SHELL_BIN_MAP` to map known binaries to specific permission keys (e.g., `git` to `shell:git`, `rm` to `shell:rm`). Any binary not found in this map falls back to a generic `shell:unknown` key with a `normal` tier.

This creates a massive vulnerability: if a user clicks "Allow for Project" on a harmless unmapped command (e.g., `grep` or `jq`), the system records `shell:unknown: allowed` in `.koincode/config.json`. Consequently, **all future unmapped commands** (including potentially dangerous execution engines like `bash`, `curl`, `python`, `node`) are automatically approved without prompting the user.

## Proposed Solution: Dynamic Scoping

Instead of trying to map the infinite universe of CLI tools, we will dynamically generate permission keys scoped directly to the binary name being executed.

### Core Architecture Changes

1. **Dynamic Key Generation:** 
   If a command is not explicitly mapped in `SHELL_BIN_MAP`, its permission key will become `shell:bin:[binary-name]` instead of `shell:unknown`.
   
2. **TypeScript Type Adjustments:** 
   The `PermissionKey` type will be updated to accept dynamic string literals using template literal types (e.g., `` `shell:bin:${string}` ``).

3. **Persistent Approvals:** 
   When a user clicks "Allow for Project" on `docker`, the system records `shell:bin:docker: allowed`. Future calls to `python` or `bash` will remain unapproved.

## Edge Case Protections

To ensure this dynamic fallback is completely secure, three additional preprocessing steps will be applied to the raw command string before attempting to extract the binary name:

1. **Subshell Detection (`shell:subshell`)**
   If the command contains `$(...` or `` `...` ``, we cannot reliably statically analyze what will be executed. These commands will immediately be classified as `shell:subshell` and elevated to the `destructive` tier to force explicit user review.

2. **Environment Variable Bypasses**
   Commands like `NODE_ENV=production rm -rf .` bypass the current naive `split(/\s+/)[0]` logic because the first word is an environment assignment. The parser will now scan past leading assignments (matching `/^[a-zA-Z_][a-zA-Z0-9_]*=/`) to find the true executable binary.

3. **Execution Interpreters**
   Even with dynamic keys, some tools are inherently dangerous because they can execute arbitrary scripts (e.g., `bash`, `sh`, `python`, `node`). We will map these explicitly in `SHELL_BIN_MAP` to the `destructive` tier so they can never be quietly auto-approved.

## Success Criteria

- Approving an unknown command like `jq` permanently allows `jq`, but attempting to run `curl` still triggers an approval prompt.
- `NODE_ENV=production cat .env` correctly triggers the `file:sensitive` destructive warning.
- `echo $(rm -rf /)` correctly triggers the `shell:subshell` destructive warning.
- Type safety for `PermissionKey` is maintained across the project configuration and CLI modules.

- Ask the agent to run echo $(ls). Verify it hits shell:subshell and requires destructive approval.
- Ask the agent to run bash -c "rm -rf /". Verify it hits the destructive tier for interpreters.
